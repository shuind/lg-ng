## 二、system-check-service 如何读取文件

`generateSystemCheckReport` 根据 `checkType` 决定读取哪些文件：

| checkType | 读取文件 |
|-----------|----------|
| `foreshadowing` | `剧情管理/伏笔清单.md`、`剧情管理/关键事件.md`、`章节摘要/*.md` |
| `timeline` | `状态追踪/时间线.md`、`剧情管理/关键事件.md`、`章节摘要/*.md` |
| `character_position` | `状态追踪/角色位置.md`、`章节摘要/*.md`、`人物设定/{target}.md`（如有） |
| `reader_experience` | `读者体验/信息差.md`、`读者体验/情绪账户.md`、`读者体验/爽点债务.md`、`章节摘要/*.md` |
| `quality` | `写作约束/禁止项.md`、`写作约束/质量约束.md`、`剧情设计指南.md`、`章节正文/*.md`（最多 3 个） |

每个文件最多读取 2000 字符，超长截断。所有文件内容拼接后传给 LLM 或规则 fallback。

## 三、每种 checkType 检查什么

### foreshadowing（伏笔检查）
- 已埋但未回收的伏笔（`⬜待回收` 标记）
- 标记为已回收但没有回收说明的伏笔
- 伏笔回收章节是否缺失
- 关键事件中是否出现疑似回收但伏笔清单没更新

### timeline（时间线检查）
- 同一事件出现多个时间
- 时间顺序和章节顺序不一致
- 没有故事时间的关键事件
- "当前章节/当前时间"缺失

### character_position（角色位置检查）
- 同一角色在相近章节出现在多个地点
- 角色位置更新缺失
- 章节摘要中出现角色但角色位置未记录
- target 存在时重点检查该角色

### reader_experience（读者体验检查）
- 读者知道的信息与角色知道的信息混乱
- 情绪债务/爽点债务长期未兑现
- 章节摘要中已有兑现但债务文件未更新
- 反转/悬念是否缺少铺垫

### quality（写作质量检查）
- 是否出现禁止项中的表达
- 是否违反质量规则
- 是否重复使用禁用比喻/套路
- 是否存在明显风格偏离

## 四、LLM 报告与 fallback 报告差异

### LLM 报告
- 结构化 Markdown：结论 → 发现的问题 → 建议的后续 ActionPlan
- 每个问题包含：类型、涉及位置、证据、风险、建议
- 能跨文件关联分析（如伏笔清单 vs 关键事件 vs 章节摘要）
- 能识别语义层面的矛盾

### Fallback 报告
- 列出读取了哪些文件
- 用正则扫描已知标记：`待回收`、`待兑现`、`未指定`、`读者未知` 等
- 列出匹配到的条目
- 明确提示"开启 LLM 以获得深度检查"
- 不做语义分析，只做文本模式匹配

## 五、ActionPlan 如何接入 system_check

1. 用户输入自然语言（如"检查伏笔有没有忘记回收"）
2. 规则 fallback 或 LLM 识别为 `system_check` 类型，带 `checkType` 字段
3. `generateActionPlan` / `generateActionPlanFromLlmActions` 生成 ActionNode，diff 中包含 `检查类型: {checkType}`
4. 用户确认后，`confirmNode` 从 diff 提取 checkType 和 target
5. 调用 `executeSystemCheck(bookId, goal, checkType, target)`
6. `executeSystemCheck` 调用 `generateSystemCheckReport` 获取真实报告
7. 报告通过 `writeBookFile` 写入 `检查报告/{timestamp}-{checkType}.md`

---

## 六、为什么不会修改正文

- `executeSystemCheck` 只调用 `generateSystemCheckReport` 获取报告内容，然后写入 `检查报告/` 目录
- 报告生成过程中只读取文件（`readBookFile`），不写入任何设定或章节文件
- `writeBookFile` 的目标路径固定为 `检查报告/{timestamp}-{checkType}.md`
- 符合 ActionPlan 设计精神：check 是"审查"而非"修改"

## 九、如何手动验证

### 1. 检查伏笔回收

输入："检查伏笔有没有忘记回收"

预期：
- 右栏出现 check 计划，diff 包含 `检查类型: foreshadowing`
- 确认后生成 `检查报告/*-foreshadowing.md`
- 报告列出未回收伏笔或说明未发现

### 2. 检查时间线

输入："检查时间线有没有矛盾"

预期：
- 生成 `检查报告/*-timeline.md`
- 报告列出时间线记录或矛盾

### 3. 检查角色位置

输入："检查林晓的位置是否混乱"

预期：
- 生成 `检查报告/*-character_position.md`
- diff 包含 `检查对象: 林晓`
- 报告重点检查林晓的位置记录

### 4. 检查读者体验

输入："检查读者信息差有没有问题"

预期：
- 生成 `检查报告/*-reader_experience.md`
- 报告列出待兑现的期待/债务

### 5. 检查写作质量

输入："检查有没有用了禁止项"

预期：
- 生成 `检查报告/*-quality.md`
- 报告列出已记录的禁止项

### 6. 无 API key

上述示例均能靠规则 fallback 工作，报告会列出扫描到的标记和文件。
