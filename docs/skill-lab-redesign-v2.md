# Skill Lab 重设计方案 v2 — 以「试验台」为核心

> 状态：框架重构，待确认分期
> 关系：本文**修正并取代 v1（`skill-lab-redesign.md`）的框架定位**；v1 里的基础设施（`skill-lab.json` sidecar、ledger `activeSkillIds` 归因、A/B 沙箱安全约束、SKILL.md 保持干净）继续沿用，不重复。
> 起因：两条关键反馈——
> 1. **高频提示词 ≠ skill**：频率可能是噪声代理，不能用它自动铸造 skill。
> 2. **探索/试验先于沉淀**：用户在沉淀之前，就需要主动探索、试验好的提示词；探索是一等公民，不是 chat 边角料。

---

## 一、v1 框架错在哪

v1 把 Lab 设计成一条**单向、自下而上**的流水线：

```
从真实使用观察 → 孵化 → 写作中用 → 重写率验证 → 沉淀为 active
```

它隐含两个错误假设，正好被上面两条反馈击中：

| v1 假设 | 为什么错 |
|---|---|
| 反复出现 + 被接受的提示词 ≈ skill | **频率是噪声代理**。「继续/写下一章/改短点」高频却零craft；「让萧炎打赢」复现却是剧情指令非原则；而真正的好 skill 可能只用过一次。频率只能产**线索**，不能产 skill |
| skill 主要从实践里「沉淀」出来 | **skill 也可以是被主动试验出来的**。作者会刻意打磨一条提示词、反复试到好用为止——这是创作的核心动作，发生在任何沉淀之前。v1 把它降级成 chat 边角料，是把主路径当成了辅路径 |

---

## 二、修正后的模型：两条来源 → 一个试验台 → 真实使用验证

```
        ┌─ 试验（自上而下·主动）── 用户打磨/试一条提示词直到好用
        │                              │
        │                              ▼
        │                        ┌──────────┐        ┌──────────┐
        │                        │  试验台   │ ─产出─▶│  Skill   │
        │                        │ (A/B 引擎)│        │(experimental→active)
        │                        └──────────┘        └──────────┘
        │                              ▲                   │
        └─ 沉淀（自下而上·被动）── 线索 ┘                   │ 真实使用
              从使用里挖出"可能值得试"的提示词              ▼
                 （线索，非 skill）              重写率（被动验证·角标）
```

三个角色，职责清清楚楚：

- **试验台（主角）**：写/改一条候选指令 → 选样本 → 沙箱跑 → A/B 比对 → 满意就存成 skill。**可空手进入**——什么都没沉淀也能直接开始打磨（满足反馈 2）。
- **沉淀线索（助手）**：从真实使用里挖出"你常这么指挥、且效果不错"的提示词，**明确标为线索而非 skill**，每条带「拿去试验」送进试验台预填。**线索 → 试验台 → skill，绝不频率 → skill**（满足反馈 1）。
- **真实使用验证（环境信号）**：skill 投入写作后，用 ledger 归因算重写率，作为 skill 上的**角标**（用了 N 次 · 重写率 X%），不单独占一屏。

### 关键区分：试验（主动）vs 验证（被动）

v1 把这两件事混成了「验证」。它们是不同的job：

| | 试验 | 验证 |
|---|---|---|
| 触发 | 用户主动发起 | 真实写作自动累积 |
| 手段 | A/B 沙箱跑（带/不带、版本A/版本B） | ledger 重写率归因 |
| 作用 | **塑造** skill（形成、打磨、迭代） | **确认** skill（站不站得住） |
| 时机 | skill 诞生之前/期间 | skill 投入使用之后 |

A/B 因此从 v1 的「冷启动探针」**升回中心**——它是试验引擎；重写率退为环境角标。

## 三、试验台详解（新的核心）

一次试验会话 `Experiment`：

1. **起点**（三种入口，都落到同一张试验台）：
   - 空手开始：用户直接写一条候选指令。
   - 从线索来：点沉淀线索的「拿去试验」，候选指令预填为挖出的提示词。
   - 改进已有 skill：把现有 SKILL.md 载入为「版本 A」，试改成「版本 B」。
2. **选样本**：默认自动挑一段相关正文（取自线索证据所在场景 / 最近章节），支持换段 / 编辑器选段 / 粘贴。
3. **沙箱跑**：`callChatCompletion`，**纯文本返回、绝不写文件**。两种比对：
   - 带 vs 不带这条指令；或 指令版本 A vs B。
4. **比对 + 迭代**：并排 + diff 高亮（复用 ledger diff 视觉）。可选 AI「解释差异」（只解释，不打分、不冒充裁判）。不满意就改指令再跑——**这是紧的创作循环**。
5. **产出**：满意 → 存成 skill（默认 `experimental`，标「实验中」，写作中可选用）。试验会话的指令版本与样本留痕，作为 skill 出处。

> 试验台同时服务"探索全新写法"和"隔离单条 skill 的效果"，把 v1 的探索支柱和 A/B 探针合并成一件事。

## 四、沉淀线索详解（降为"产线索"，不再"产 skill"）

挖掘逻辑（沿用 v1 的归因地基，但**输出语义降级**）：

- 数据：chat 提示词（`role="user"`）+「提示词→产出→是否被接受」归因 + 手改聚类（diff 仅佐证）。
- **分层过滤**（频率与 craft 判断都要，叠加而非二选一）：
  1. **粗筛（频率 + 接受度）**：先按语义聚类找反复出现、且产出被接受的指令簇——这一层只负责"缩小候选池"，廉价、无模型。
  2. **精筛（LLM 判断 craft）**：对粗筛出的候选再用 LLM 过滤掉**操作噪声**（继续/改短）与**情境专属**（让某角色赢），只留**规范性、可泛化、值得自动生效**的指令。
  3. **蒸馏**：把情境化指令抽象成可复用规则（去人名/场景）。
  > 频率负责"省钱地缩小范围"，LLM craft 判断负责"把住质量关"。少了第 2 层，高频噪声会灌进来（反馈 1）；少了第 1 层，每条提示词都过模型太贵。
- 输出 = **线索（lead）**，不是 skill。线索卡片话术：「你已 6 次这样指挥 AI、5 次产出直接采用——**要不要试验成 skill？**」按钮是 `[拿去试验]`，不是 `[直接生成]`。
- 线索持久累积（strength 增长、证据去重、可忽略），但**成为 skill 的唯一出口是试验台或用户显式确认**。

## 五、UI（极简收敛）

Lab 本质 = **一个试验台 + 一条线索流**，验证是 skill 角标。

```
 Skill Lab                                              从最近使用找线索 ↻
 ┌─────────────────────────────┬───────────────────────────────────┐
 │  试验台（主区）               │  线索流（侧栏）                     │
 │  指令: [____________________] │  · 你已6次这样指挥·5次直接采用       │
 │  样本: [自动挑/换段/粘贴]      │    "战斗先交代力量差..."  [拿去试验] │
 │  [跑 A/B]                     │  · 你已4次要求对话留潜台词           │
 │  ┌── A 不带 ──┬── B 带 ──┐    │    "别把情绪写白..."     [拿去试验] │
 │  │  ...       │  ...      │    │  · (操作噪声/情境专属已被过滤掉)    │
 │  └───────────┴───────────┘    │                                   │
 │  [存成 skill (实验中)]         │                                   │
 └─────────────────────────────┴───────────────────────────────────┘
 已有 skill 列表：每张带验证角标「用了12次·重写率25%·实验中」
```

- 空手可用：不点线索也能直接在试验台写指令开跑。
- 线索只能「拿去试验」，点完跳进左侧试验台预填——杜绝「频率直接变 skill」。
- 验证（重写率）是 skill 卡片/选择器上的角标，不占独立视图。

## 六、数据模型增量（相对 v1）

```ts
// 线索：明确是 lead，不是 skill（沿用 v1 Observation，强调语义）
interface Lead extends Observation {
  isCraft: boolean            // LLM 判断：规范可泛化，非操作噪声/情境专属
  distilledRule: string       // 蒸馏出的可复用规则
  // 唯一出口：sendToBench() → 试验台，或用户显式 confirm
}

// 试验会话：v1 的 Trial 升级为一等公民、支持迭代
interface Experiment {
  id
  entry: "scratch" | "from_lead" | "improve_skill"
  instructionVersions: string[]      // 多次迭代的指令版本
  sample: { source: "auto"|"editor"|"paste"; text }
  runs: { instruction; output; mode: "with_without"|"a_b" }[]
  outcome?: { savedSkillName: string }  // 满意则产出 skill
}

// skill 履历：验证退为角标（沿用 v1 SkillUsageStats + stage）

// SkillLabMeta.usage: { timesUsed, timesRewritten, rewriteRate }
// SkillLabMeta.originExperimentId / originLeadId  ← 出处可追溯
```

## 七、构建块（对等，不强制先后）

不再排成 1→2→3 的死序。试验台与沉淀线索是**对等来源**，谁先做都行，可并行也可单独验收。只有一个**极小的共享地基**，其余三块各自独立。

### 地基（很小，任何一块开工前先落）
- `skill-lab.json` sidecar 扩展（沿用 v1）：`SkillLabMeta.stage = experimental | active`。
- 「实验中」角标（skill 卡片 / chat 选择器）。

### 块 A — 试验台（自包含，零外部依赖）
- 新增 `skill-trial-service.ts`：A/B 沙箱 completion（纯文本、不写文件）+ 迭代。
- `app/api/.../skills/lab/experiment` 等端点。
- 试验台 UI：指令编辑 + 样本选择 + A/B 并排 diff + 「存成 skill（实验中）」。
- **特点**：不依赖使用积累，空项目当天可用。最适合作为最先能"摸到"的成果，但不强制。

### 块 B — 沉淀线索（依赖归因地基）
- ledger `activeSkillIds` 归因地基（chat-service 透传 `selectedSkills`）。
- `skill-lab-service.ts`：读使用 → 分层过滤（频率+接受度粗筛 → LLM craft 精筛 → 蒸馏）→ 产**线索**；累积合并。
- 线索流 UI + 「拿去试验」预填试验台（与块 A 对接）。

### 块 C — 真实使用验证（依赖归因地基，与块 B 共享）
- 归因聚合 → `SkillUsageStats` 重写率。
- skill 角标「用了 N 次·重写率 X%」；高重写率 → 反哺线索「这条 skill 该改」。

> 依赖关系：A 独立；B 与 C 共享「归因地基」（`activeSkillIds`），建议 B/C 一起做。三块之间无强制先后，按你想先看到哪个价值来排。

## 八、贯穿原则（在 v1 基础上修订）

- **试验台是主角**：探索/试验是一等创作动作，可空手进入，先于任何沉淀。
- **频率只产线索，不产 skill**：唯一成 skill 的出口是试验台或用户显式确认。
- **线索分层过滤**：频率+接受度粗筛（省钱缩范围）叠加 LLM craft 精筛（去操作噪声/情境专属），两层都要。
- **试验（主动塑造）≠ 验证（被动确认）**：A/B 是试验引擎，重写率是环境角标。
- **两条来源对等**：试验台与沉淀线索无强制先后，按价值优先级排。
- **真实 AI，不造假指标**：AI 只做 craft 判断/蒸馏/解释差异，不打分、不冒充裁判。
- **沙箱安全**：试验只读 + 纯文本返回，绝不写项目文件。
- **可追溯出处**：每条 skill 回指它的试验会话 / 线索 / 手改证据。
- **极简 UI**：试验台 + 线索流双栏，验证是角标，不堆 dashboard。
