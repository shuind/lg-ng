## 三、LLM schema 如何扩展

Zod schema 从 2 个 variant 扩展到 6 个：

```ts
z.union([
  z.object({ type: z.literal("gender_change"), ... }),
  z.object({ type: z.literal("relationship_change"), ... }),
  z.object({ type: z.literal("character_create"), name, fields? }),
  z.object({ type: z.literal("character_update"), character, field, value }),
  z.object({ type: z.literal("world_update"), fileHint, section?, content }),
  z.object({ type: z.literal("chapter_check"), chapterHint, target?, checkGoal }),
])
```

Prompt 在"支持的动作类型"部分新增了 3-6 四种类型的说明和 JSON 示例。原有类型 1-2 的描述不变。

## 四、规则 fallback 支持哪些句式

### character_create

- "新建人物赵霜"
- "创建角色林晓"
- "新增人物赵霜"
- "创建一个名叫星城守卫的人物"

正则：`/(?:新建|创建|新增)\s*(?:一个|一名)?\s*(?:人物|角色)\s*(?:叫|名叫)?\s*(\S+)/g`

### character_update

- "把林晓的身份改成禁军斥候"
- "林晓的年龄改成24岁"
- "将陈磊的性格改成阴沉"

正则：`/(?:把|将)?\s*(\S+)\s*(?:的)?\s*(性别|年龄|身份|外貌|性格|背景|目标|备注)\s*(?:改|换|变成?)\s*(\S+)/g`

### world_update

- "在世界观里增加一个城市星城"
- "世界观新增灵石设定"

正则：`/(?:在世界观里|世界观)\s*(?:增加|新增|添加|修改)\s*(?:一个|一条)?\s*(\S+?)(?:[：:]\s*(.+))?$/g`

### chapter_check

- "检查第二章有没有和林晓设定冲突"
- "扫描第三章的人物一致性"

正则：`/(?:检查|扫描|查看)\s*(第.{1,3}章|.+?章)\s*(?:有没有|是否存在|是否)\s*(.+?)(?:冲突|矛盾|一致|设定)?$/g`

## 六、confirm 后如何执行

### character_create

1. 检测：`operation === "create" && scopePath.startsWith("人物设定/")`
2. 文件不存在 → 创建模板（fullwidth space `　` 分隔符，兼容 setting-card-store 解析）
3. 文件已存在 → 追加 `## 补充设定` section
4. 调用 `writeBookFile()` → dirty + Ledger 自动生效

### character_update

1. 检测：`operation === "update" && scopePath.startsWith("人物设定/") && diff.includes("→") && !diff.includes("性别")`
2. 文件不存在 → 创建基础文件 + 写入字段
3. 对 `性别/年龄/身份`：替换 `**field**　value` 行
4. 对其他字段：找 `## {field}` section 替换，找不到追加新 section
5. 调用 `writeBookFile()`

### world_update

1. 检测：`operation === "update" && scopePath.startsWith("世界观/")`
2. 在 `世界观/` 下扫描匹配文件名，找不到则创建
3. 如果有 section → 找 `## {section}` 追加内容，找不到追加新 section
4. 如果无 section → 追加 `## 补充设定`
5. 调用 `writeBookFile()`

### chapter_check

1. 检测：`operation === "check"`
2. 读取匹配的章节正文
3. 如果有 target → 读取相关设定文件
4. 如果 LLM 可用 → 调用 LLM 生成冲突检查报告
5. 无 LLM → fallback 报告（列出已检查文件 + 提示人工确认）
6. 报告写入 `检查报告/{timestamp}-{chapterHint}.md` 通过 `writeBookFile()`
7. 不修改章节正文

---

## 七、chapter_check 为什么不直接改正文

1. 设计意图是"审查"而非"修改"——用户期望是"告诉我有没有问题"
2. 直接改正文风险高，AI 改错难以回滚
3. 符合 ActionPlan 确认机制的精神：先看计划，再决定
4. 生成的检查报告文件通过 writeBookFile 写入，同样有 dirty + Ledger 记录

## 八、dirty / Ledger 是否生效

是。所有 4 种新动作的文件写入都通过 `writeBookFile()`，因此：

- `markDirty()` 自动调用 → dirty-files.json 更新
- `appendLedgerEntry()` 自动调用 → ledger.jsonl 有 before/after 快照记录
- `book.json` 的 updatedAt 自动更新

输入："新建人物赵霜，女，身份是北境医师"

- 右栏出现 create 计划，scopePath = `人物设定/赵霜.md`

输入："把林晓的身份改成禁军斥候"

预期：
- 右栏出现 update 计划
- 确认后 `人物设定/林晓.md` 的 `**身份**` 行变化

### 3. 更新世界观

输入："在世界观里增加一个城市星城"

预期：
- 右栏出现 update 计划，scopePath = `世界观/星城.md`
- 确认后 `世界观/星城.md` 被创建

### 4. 检查章节

输入："检查第二章有没有和林晓设定冲突"

预期：
- 右栏出现 check 计划
- 确认后不修改章节正文
- 生成 `检查报告/*.md`
- Ledger 有报告写入记录

### 5. 无 API key

上述示例至少部分能靠规则 fallback 工作（character_create、character_update、world_update、chapter_check 的正则匹配）。
